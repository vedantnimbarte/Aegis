# Technical Specification Document: AutoPentest SaaS

## 1. System Architecture Overview
The AutoPentest platform follows a decoupled, service-oriented architecture designed to handle long-running background tasks (Strix AI pentesting) asynchronously while maintaining a highly responsive web frontend.

* **Frontend:** Next.js (React), Tailwind CSS, shadcn/ui.
* **Backend API:** FastAPI (Python) for seamless integration with the Strix Python ecosystem.
* **Database:** PostgreSQL for persistent relational data.
* **Job Queue:** Redis + Celery for managing asynchronous Strix scanning jobs.
* **Execution Environment:** Docker Engine API for spinning up isolated ephemeral containers for each scan.

## 2. Database Schema (PostgreSQL)
The relational database will store user data, authentication state, repository metadata, and scan results.

### Users Table
| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | Unique user identifier. |
| `email` | VARCHAR | Unique, Not Null | User's email address. |
| `github_token` | VARCHAR | Encrypted | OAuth token for GitHub API access. |
| `subscription_tier` | VARCHAR | Default: 'free' | Current SaaS tier (starter, pro, enterprise). |
| `stripe_customer_id`| VARCHAR | Nullable | Stripe billing reference. |

### Repositories Table
| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | Unique repository record. |
| `user_id` | UUID | Foreign Key | References `Users.id`. |
| `github_repo_id` | VARCHAR | Not Null | Remote GitHub repository ID. |
| `name` | VARCHAR | Not Null | Format: `owner/repo_name`. |
| `url` | VARCHAR | Not Null | URL of the repository. |

### Scans Table
| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | Unique scan execution identifier. |
| `repository_id` | UUID | Foreign Key | References `Repositories.id`. |
| `status` | VARCHAR | Not Null | `pending`, `running`, `completed`, `failed`. |
| `scan_mode` | VARCHAR | Not Null | `quick`, `standard`, `deep`. |
| `started_at` | TIMESTAMP | Nullable | Scan start time. |
| `completed_at` | TIMESTAMP | Nullable | Scan completion time. |

### Vulnerabilities Table
| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | Unique vulnerability finding. |
| `scan_id` | UUID | Foreign Key | References `Scans.id`. |
| `severity` | VARCHAR | Not Null | `critical`, `high`, `medium`, `low`. |
| `title` | VARCHAR | Not Null | Short title (e.g., "SQL Injection in /login"). |
| `description` | TEXT | Not Null | Detailed Strix output and reproduction steps. |
| `poc_code` | TEXT | Nullable | Working Proof-of-Concept code generated by Strix. |
| `remediation` | TEXT | Nullable | Patch/fix suggestions. |

## 3. Core API Endpoints (RESTful)
The backend API will be versioned at `/api/v1/`.

| Endpoint | Method | Auth Required | Description |
| :--- | :--- | :--- | :--- |
| `/auth/github` | POST | No | Handles GitHub OAuth callback and issues JWT. |
| `/users/me` | GET | Yes | Returns current user profile and subscription status. |
| `/repos` | GET | Yes | Lists authorized GitHub repositories. |
| `/repos` | POST | Yes | Syncs/adds a repository to the user's dashboard. |
| `/scans` | POST | Yes | Triggers a new Strix scan for a specific repository. |
| `/scans/{id}` | GET | Yes | Returns status and metadata of a specific scan. |
| `/scans/{id}/report`| GET | Yes | Returns detailed vulnerabilities and PoCs for a scan. |

## 4. Strix Execution & Worker Queue Lifecycle
Because penetration testing takes time (minutes to hours depending on the codebase), it must not block the main HTTP thread. 

### Worker Flow
1. **Trigger:** User calls `POST /scans`. API creates a `Scans` record with status `pending`.
2. **Queue:** API pushes a job payload `{"scan_id": "uuid", "repo_url": "...", "mode": "quick"}` to the Redis queue.
3. **Execution:** A Celery worker picks up the job, updates the database status to `running`, and initiates a subprocess interacting with Docker.
4. **Isolation:** The worker pulls the target repository into a temporary directory and mounts it to a fresh Strix Docker container.
    * Command: `docker run --rm -v /tmp/repo:/app -e STRIX_LLM="..." usestrix/strix -n --target /app`
5. **Ingestion:** Upon completion, the worker parses the Strix output JSON file (`strix_runs/<run-name>/report.json`).
6. **Storage:** Worker maps the findings into the `Vulnerabilities` table, updates the `Scans` status to `completed`, and cleans up the temporary Docker volume.

## 5. Security & Authentication Constraints
* **Authentication:** Stateless JWTs (JSON Web Tokens) with a short expiration (15 minutes) and a refresh token mechanism.
* **GitHub Tokens:** OAuth tokens must be encrypted at rest in the PostgreSQL database using AES-256-GCM. 
* **Tenant Isolation:** All API endpoints must strictly validate that the `user_id` extracted from the JWT matches the owner of the requested `repository_id` or `scan_id`.
* **Container Security:** Strix Docker containers must run in an isolated network without access to the host's internal metadata or backend databases. Network egress should only allow HTTP/S traffic to the target LLM API and necessary OSINT endpoints.

## 6. Deployment & Infrastructure Pipeline
* **Hosting:** 
  * Frontend: Vercel (for edge caching and optimized React delivery).
  * Backend & Workers: AWS ECS (Elastic Container Service) or DigitalOcean App Platform.
  * Database: Managed PostgreSQL (e.g., AWS RDS or Supabase).
* **CI/CD:** GitHub Actions configured to run unit tests and automatically deploy to staging on `main` branch merges.
* **LLM Key Management:** Shared LLM provider API keys (OpenAI, Anthropic) injected securely via environment variables (AWS Secrets Manager) to the worker nodes.
